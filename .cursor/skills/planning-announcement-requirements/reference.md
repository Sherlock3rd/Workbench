# 公告需求生成参考

## 更新规模判断

先判断玩家能感知到的更新规模，再选择公告形态。

| 类型 | 判断标准 | 推荐形态 |
| --- | --- | --- |
| 小优化 | 单个系统内的轻量体验优化、排序优化、入口优化、提示优化 | `系统名优化：一句话说明变化和收益。` |
| 较大系统优化 | 同一系统内多处规则、任务、排行榜、匹配、教程或交互调整 | `系统名优化：` 加编号列表 |
| 大系统/活动上线 | 新增活动、新增大功能、包含多个子玩法或活动模块 | 氛围开头 + 活动名 + 子玩法列表 + 必要开放说明 |
| 赛季更新 | 新剧本、新赛季、新文明、新统帅或赛季预热活动 | 赛季亮点开头 + 预热活动 + 剧本特色玩法 |
| 规则说明型公告 | 重点是解释购买次数、参与资格、结算、匹配、排行榜、积分、移民、赛季时间等规则如何计算 | 背景原因 + 关键定义 + 计算规则 + 条件分支 + 限制例外 |

判断时只看公告受众能理解的更新，不按研发工作量判断。后台重构很大但玩家无感时，仍按小优化或不进入公告。

## 文案原则

- 面向玩家，优先表达“新增了什么、优化了什么、现在可以怎么做”。
- 保留项目既有称谓和语气。ROK 中玩家称谓优先使用术语库或既有公告口径，例如 `执政官`。
- 不承诺未确认结果，例如“必定获得”“大幅提升”“所有地区同时开启”等。
- 不写内部术语，例如需求、配置、接口、埋点、灰度、上线节点、开发中。
- 奖励、时间、开放范围若文档未明确或可能运营调整，使用“详见活动日历 / 以游戏内实际开放为准 / 待确认”。
- 对多条优化进行合并时，优先按玩家体验链路排序：入口与参与方式、核心玩法、规则任务、奖励结算、排行榜或匹配。

## 活动类公告必读信息

活动、赛季、追赶、回归、节日、运营投放、商店兑换、BP、转盘、排行榜等内容，不能只概括为“获取更多奖励”或“提升养成进度”。生成公告前必须逐项读取并梳理：

| 信息 | 必须回答的问题 | 公告写法 |
| --- | --- | --- |
| 背景/目的 | 为什么这次要新增或优化？玩家当前痛点是什么？ | 用玩家视角解释，例如“帮助新晋执政官更快补足征服赛季养成差距”，不要写“提升付费 ROI”“解决投放缺口”等内部目的 |
| 参与对象 | 谁能参加？全体玩家、指定赛季、指定王国、回归玩家、新晋顶 R、中小 R、国王或联盟成员？ | 写清“符合条件的执政官”“S3 赛季执政官”“参与报名王国的国王”等，不确定则列入待确认 |
| 开放节点 | 何时开启？随哪个赛季、活动阶段、版本更新、报名期或 K 前夕开启？ | 可写“将在 S3 赛季开放”“随 K3 前夕活动开启”，未确认具体时间不要编日期 |
| 持续时间 | 活动、本体、BP、商店、预热、兑换期分别持续多久？ | 有明确数值必须写，例如“活动持续 51 天，配套 BP 持续 49 天” |
| 参与方式 | 玩家要做什么？抽转盘、购买礼包、完成 BP、兑换商店、投票、竞猜、完成任务？ | 用动作表达，不写“相关链路”“功能模块”等内部说法 |
| 参与次数/购买上限 | 每日次数、总次数、礼包上限、抽取次数、兑换上限、折扣次数是多少？ | 关键数值必须进入公告；如“100 美元礼包购买上限提升至 3 次”“每日宝石购买次数增加 90 次” |
| 奖励/兑换内容 | 能获得什么？货币、图纸、觉醒石、宝箱、统帅碎片、金头、赛季币等 | 用玩家能识别的道具名；术语未确认时保留原文并待确认 |
| 解锁/分层规则 | 商店、巅峰奖励、排行榜、任务是否有解锁条件？ | 写清“普通商店奖励全部购买后解锁巅峰商店”等条件 |
| 动态规则 | 购买数量、匹配、排行榜、兑换上限是否随玩家进度变化？ | 说明动态依据的玩家可理解口径，例如“根据当前培养进度动态计算”，不展开公式 |
| 结算/回收 | 活动结束后积分、货币、奖励如何处理？ | 只有明确且对玩家重要时写；标注待定的比例不得进入正式公告 |
| 限制/例外 | 移民、跨赛季、开放地区、王国开放天数、赛季差异有什么影响？ | 对参与资格有影响的必须写；内部兜底逻辑不写 |

生成活动公告前，先内部列出“参与对象 / 开放节点 / 持续时间 / 参与次数 / 核心奖励 / 解锁条件 / 限制例外”。缺少会影响玩家理解的信息时，不要用泛泛的“更多奖励”“海量豪礼”替代，应写入 `待确认事项`。

## 规则说明型公告

如果方案核心在解释“怎么算、谁有资格、什么时候有次数、什么情况下不能参与”，必须按规则说明型公告处理。不要写成活动宣传稿。

### 识别信号

出现以下信息时，优先判断为规则说明型公告：

- `可购买次数`、`购买上限`、`兑换上限`、`次数计算`、`不计入`。
- `注册时间`、`开服时间`、`进入征服赛季时间`、`周期`、`完整周期`。
- `上线时间晚于/早于`、`首次进入`、`活动开启前/后`。
- `同赛季移民`、`跨赛季移民`、`未开启活动的王国`、`无法继续参与`。
- `共享上限`、`动态计算`、`按差值计算`、`达到最高进度`。

### 必须提炼的结构

| 结构 | 提炼目标 | 写法要求 |
| --- | --- | --- |
| 背景原因 | 玩家为什么需要知道这条规则？这次解决什么公平性、理解成本或参与资格问题？ | 第一句先讲玩家可理解原因，例如“为确保不同王国赛季节奏下可购买次数的公平与一致性” |
| 核心规则对象 | 本次到底在说明哪个规则？例如装备冲刺可购买次数，而不是整个追赶活动 | 标题或首段点明“本次活动在计算可购买次数时……” |
| 关键定义 | 规则依赖哪些时间点或名词？ | 先定义再计算，例如先定义“进入征服赛季时间” |
| 条件分支 | 不同情况下分别怎么算？ | 用“若……则……”拆条，不合并成长句 |
| 计算公式 | 周期、次数、比例、上限如何计算？ | 公式必须转成自然语言，保留关键数值 |
| 不计入/无资格 | 什么情况下没有次数、不能参与、不会继续生效？ | 负向规则必须明确写出，不能用“以游戏内为准”代替 |
| 共享/动态规则 | 多个道具、图纸、商店是否共享上限或动态计算？ | 写清共享范围和动态依据，不展开内部公式 |
| 移民/赛季限制 | 移民、跨赛季、未开放王国如何影响活动？ | 对玩家参与资格有影响时必须进入公告正文 |

### 输出顺序

规则说明型公告优先使用以下顺序：

1. 背景和目的：为什么要统一或调整这条规则。
2. 关键定义：先解释计算基准，例如进入征服赛季时间。
3. 计算规则：周期、次数、比例、上限。
4. 道具/奖励差异：不同图纸、道具、商店是否有不同规则。
5. 限制例外：无次数、无法参与、移民后结束等。

### 禁止写法

- 不要把规则说明写成“全新活动即将开启，赢取海量奖励”。
- 不要只写“购买次数将根据进度动态计算”，必须说明动态依据或条件。
- 不要省略“不足 1 个周期不计入”“无可购买次数”“无法继续参与”等负向条件。
- 不要把内部背景照搬成公告，例如“运营过程中多次调整”可以保留，但“CP 目标、ROI、付费行为、数据验证”不能写。

### 示例结构

```text
为确保不同王国赛季节奏下可购买次数的公平与一致性，本次活动在计算可购买次数时，将采用统一标准：

执政官进入征服赛季时间的判定：
1. 若注册时间在注册服进入征服赛季之后，则以注册时间作为进入征服赛季时间。
2. 若注册时间在注册服进入征服赛季之前，则以注册服进入征服赛季时间作为进入征服赛季时间。

可购买次数计算：
1. “失落之地”周期将固定为 100 天。
2. 每落后 1 个完整周期，可获得 2 次可购买次数；不足 1 个完整周期不计入。
3. 不同图纸的可购买数量与其加入征服赛季的时间相关，但共享同一可购买次数上限。
4. 若图纸上线时间晚于执政官进入征服赛季时间，则无可购买次数。

“装备冲刺”活动仅在进入征服赛季的王国开启。若在活动期间移民到未开启此活动的王国，则无法继续参与此活动。
```

### 优化活动的背景写法

优化活动不是只写“优化了某规则”。必须读取 `设计目的`、`背景`、`问题`、`玩家反馈`、`线上数据`、`讨论` 等相关内容，判断这次优化要解决的玩家问题。

- 如果背景是玩家看不懂规则，公告里写“为帮助执政官更清晰了解……”
- 如果背景是追赶困难，公告里写“帮助符合条件的执政官更快补足……”
- 如果背景是入口/排序/展示低效，公告里写“现在可以更快找到……”
- 如果背景是安全或误操作，公告里写“为提升账号/资源操作安全性……”
- 如果背景是奖励投放不足，公告里写“增加了……获取/购买/兑换机会”，不要写内部投放、营收或 ROI

背景说明只写玩家能感知的原因，不写“业主群反馈”“CP 目标”“付费行为”“ROI”“数据验证”“研发改造”等内部来源。

## ROK 润色列沉淀

以下规则来自 ROK 版本公告内容表历次版本页签的 `润色（文案）`、`润色` 及早期参考公告正文。该列常包含文案同学 @ 信息，提炼样式时必须忽略 @ 对应人，只读取实际公告正文。

### 读取口径

- 必须先读表头，不按固定列号取值。近期页签通常是 `润色（文案）` / `润色（AI）`，部分页签只有 `润色`，早期参考页签可能把正式公告正文放在 `备注/解释/截图` 列。
- 权威优先级：`润色（文案）` > `润色` > 已发布参考公告正文 > `润色（AI）`。`润色（AI）` 只用于观察可能的表达方向，不直接作为正式公告口径。
- 同一行有文案润色和 AI 润色时，优先学习文案润色的短版、保守、准确写法；AI 润色中更营销、更长或带 HTML 标记的内容不要直接复用。
- 历史润色只用于学习写法，不作为新需求事实来源。活动时间、范围、奖励、限制仍以当前策划案为准。
- 遇到国服/国际服差异时，按历史公告做法拆成两句或括号标注，例如“（国服）”“（国际服）”，不要混成一个无差异口径。

### 基础润色方向

- 能一句话讲清的小更新，不强行加标题；直接写成完整陈述句，例如“装备图纸和图纸碎片的详情描述中添加了对应道具的持有数量信息。”
- 安全、验证、权限类更新优先写“增加了/添加了……所需的……”，说明触发操作和新增保障，例如“增加了军器铭刻和转换操作所需的二级密码验证。”
- 体验表现类更新可用“系统名/表现名优化：”加一句说明，例如“行军特效优化：增加了大地图上部队行军特效的渐入和渐隐效果。”
- 排序、展示、列表优先突出“现在/新增后玩家能更快做什么”，例如“现在获取转换石碎片的委托会显示在更优先的位置。”
- 润色时删除策划案中的重复动词和内部表达，把“新增了/优化了/简化了”统一成玩家能理解的结果，不解释技术原因。
- 文字描述修正、文本优化、显示修复必须说明是否影响实际效果。例如“修正了部分语种描述，不影响实际生效逻辑”“技能实际表现与机制没有改变”。
- 规则调整要保留前后变化和适用范围，优先使用“由……调整为……”“将在……开启先锋体验”“仅在……生效”等表达，不用“规则更合理”等泛化表述。
- 入口/权限/解锁类功能要写清谁能使用、从哪里进入、什么条件解锁。例如“联盟盟主、顾问和有权限的官员可以……”“该功能会在联盟所在王国进入第1赛季后解锁”。
- 体验优化类文案可补充玩家收益，但不能夸大。常用落点包括“使调整更加便捷”“提升战场体验”“更容易被点击选中”“可通过录像进行复盘”。
- 将策划口语改成公告口径：`玩家` 优先改为 `执政官`，`R4/R5` 改为 `盟主、顾问和有权限的官员`，`灰度 / beta / 测试` 改为 `先锋体验`，`BOSS` 改为 `首领`，`服务器` 优先改为 `王国`。
- 删除内部语气和不必要解释，例如“其实是个线上 bug 修复”“不要写成砍效果”“后续假设版本更新时间调整”等，只保留玩家需要知道的结果和范围。
- 不照抄 AI 润色中的过度营销收尾，例如“请执政官们关注游戏内公告，以获取更多详细信息”“享受更为优化的游戏体验”等泛化句，除非历史文案列也采用。

### 多条优化写法

- 标题使用 `<活动/系统名>优化：`，下方用 `1.`、`2.` 编号。
- 每条只表达一个玩家可感知变化，避免一条中混入多个系统点。
- 规则类写“优化了……规则，目前……”，任务类写“优化了……，执政官在……即可……”，榜单类写“优化了排行榜，将……合并成为……”。
- 教程和规则展示可作为独立条目，写清新增位置和用途，例如“在规则界面新增地图介绍页签，执政官可通过该页签了解地图元素与玩法。”
- 涉及王国、联盟、执政官等 ROK 固定称谓时，优先使用公告口径，不使用“玩家”等泛称替换已确认称谓。
- 如果优化点天然按模块拆分，可以使用短标题承接，例如“目标扩展：”“交互增强：”“操作自定义：”“位置优化：”。但正式版本公告通常优先使用编号，不强行写成宣传页结构。
- 大段复杂系统优化按玩家体验链路排序：参与/入口 -> 规则/条件 -> 战斗或玩法变化 -> 奖励/结算 -> 展示/界面。
- 修复类内容如果只是单点，写成一句“修复了……的问题”；只有同一系统多条修复时才合并成“<系统>相关修复：”并列点。
- 复杂玩法改动中若有“新增、调整、修复”混杂，不要全部写成“优化”。按事实拆分为“新增…… / 调整…… / 修复……”，避免让玩家误判强度变化。
- 若一条公告涉及多个数值或条件变化，保留玩家可核对的数值，不用“若干优化”“体验提升”代替。例如开放期、保护期、可购买次数、每日上限、活动阶段和排行榜范围。

### 活动上线写法

- 大活动可用一句氛围开头，再接“<活动名>活动即将开启，我们准备了精彩的……等您参加：”。
- 子活动使用 `1）`、`2）` 编号，格式为 `<子活动名>：<一句话卖点>！`。
- 节日活动、运营活动可以强化情绪词，如“战火再燃”“夏日惊喜”“赢取海量豪礼”，但不要替换已确认活动名和道具名。
- 活动开放时间、地区、赛季奖励差异等限制统一放在末尾 `*` 说明，不插入每个子活动条目中。
- 如果活动围绕现实赛事或主题玩法，开头先交代主题和参与收益，再列核心模块，例如竞猜、商店、联盟竞答。
- 如果新增活动会替代或取消旧活动，短版版本公告必须写清替代关系，结构优先为“新增<场景>活动<活动名>：活动将在<范围/节点>开启，同时取消原有的<旧活动>；执政官可通过<核心行为>获得<积分/货币>，并根据<排名/阶段/结算规则>获取奖励。”不要展开预告期、展示入口、排名公示等详情，除非这些信息本身是玩家必须提前知道的变化。
- 先锋体验用“<功能/活动>开启先锋体验！”或“将在……开启先锋体验”承接，不写成全量上线；必须保留“部分王国 / 部分失落之地 / 版本更新后新开启的剧本 / 正在进行或报名中的失落之地不受影响”等范围限制。
- 常驻开放类活动或系统优先写“<系统>开放常驻：”或“<系统>常驻开放”，随后写开放条件、入口和主要用途。
- 节日活动可保留一句氛围化开头，但子活动卖点仍要具体到玩法动作或奖励来源；限制统一放在末尾 `*`，例如开放时间、王国开放天数、赛季奖励差异。
- 外观/装扮/皮肤类新增需要写清获取方式、包含的外观资源、使用条件，并明确“不附加额外的数值属性加成”。
- 活动返场或数值优化返场要写清“数值优化后返场”，不要写成全新活动；若活动结束后道具会置换、回收或不回收，必须保留。
- 活动奖励替换要写清替换对象，例如“建筑加速道具奖励将直接替换为治疗加速道具”，不要只写“奖励优化”。

示例：

```text
新增失落之地活动“中期结算”：
- 活动将在所有征服赛季的中期开启，同时取消了原有的“清剿敌军”活动；
- 执政官可以通过占领圣地、击杀敌军、治疗伤兵等行为获得个人积分和军团积分，并根据个人积分排名获取丰厚奖励。
```

### 1108-1092 样本修正规则

以下来自版本公告内容表 `1108` 至 `1092` 页签的重点差异：

- 权限称谓：策划案中的 `R4/R5` 不直接进入公告，改为玩家可理解的权限身份，例如“联盟盟主、顾问和有权限的官员”。
- 测试范围：`灰度`、`beta 测试`、`部分服务器` 等内部或非 ROK 口径，统一改为“部分王国 / 部分失落之地开启先锋体验”。如功能可能与正式版不同，可写“先锋体验版可能与最终正式版有一定差异”。
- 术语纠偏：`贡献值` 在失落之地战斗场景中需按正式口径写为 `赛季功勋值 / 赛季功勋`；`楔形阵2.0` 等名称要按术语库或最终文案确认，不能自行统一。
- 实际效果边界：技能描述、文本、显示、图标背景等修正必须写“不影响实际效果 / 对道具功能无影响 / 仅修改显示”，避免玩家理解为数值调整。
- 版本范围：若写“版本更新后新开启的失落之地中生效”“正在进行或报名中的失落之地不受影响”，必须完整保留，不压缩为“版本更新后生效”。
- 国服/国际服：涉及 PC/桌面版本、至尊特权/SVIP、联赛赛季等差异时，按国服和国际服分别写，不混用名称。
- 任务/奖励替换：如“黄金钥匙替换为君王之匙”“建筑加速替换为治疗加速”“清剿调整为捐献加速”等，必须保留替换前后对象。
- 活动条件：如“仅在开放时间大于 60 天的王国中生效”“第 1 赛季及之后解锁”“进入征服赛季后开启”等，必须进入正文或末尾说明。
- AI 润色反例：带 `<size=30>` 标记、过度分段、泛化鼓励语、将短公告扩成宣传稿的内容，不作为最终公告风格。

### 先锋体验/赛季机制写法

- 新机制试运行优先写“<功能名>开启先锋体验！”作为标题，既保留新鲜感，也避免承诺全量开放。
- 投票、报名、跨王国等复杂规则应按条件拆条：先写何时触发、谁能操作、达到什么条件开启；再写分线、计票范围或适用对象。
- 涉及版本后新开启剧本、报名结果展示期、支持开启王国比例等限制时，必须保留条件，不改写成全服无条件开放。
- 如果内容影响正在进行的失落之地或赛季报名，必须明确“版本更新后新开启”“正在进行或报名中的不受影响”等边界。
- 复杂赛季机制可使用“在<剧本/赛季>中，<更新内容>。本次改动将在<范围>中采用先锋体验的形式上线。”的结构。

### 规则与限制写法

- 数值、时间、范围变化必须写原值和新值，例如“圣城开放争夺的时间由 8 小时缩短为 4 小时”“创建天数限制由 240 天调整为 150 天”。
- 奖励和任务调整要写清取消了什么、替换成什么、总收益是否变化。例如“取消每日挑战及其可获得经验，调整 7 日挑战的任务内容及经验，每周可获得的总经验数不变。”
- 回收、关闭、失效、仅可见、仅生效等限制要进入正文或末尾说明，不用“请以游戏内为准”完全替代。
- 仅国服、仅国际服、部分王国、部分失落之地、进入第 N 赛季后开启等范围限制，必须保留在公告文案中。
