# 策划案阅读与公告需求

## 必读前置

在执行本 Skill 前，先读取并遵守：

1. `rules/rules.md`：项目规则、回复前缀、确认边界、会话记录要求。
2. `mistakes/`：先检索相关错题，尤其是图片读取、输出格式、中间交付来源误读。
3. 本目录 `reference.md`：更新规模判断、ROK 润色列沉淀、输出模板和待确认边界。**必须实际读取，不得只读本 Skill 主文件。**
4. `C:/Users/liuweichen/.agents/skills/lark-shared/SKILL.md`：身份、权限、scope、安全规则。
5. `C:/Users/liuweichen/.agents/skills/lark-doc/SKILL.md`：飞书文档 URL/token 处理、`docs +fetch`、`docs +update`、媒体下载限制。
6. 如遇电子表格、术语库或公告内容表，读取 `C:/Users/liuweichen/.agents/skills/lark-sheets/SKILL.md`。
7. 如需写入、追加、替换或格式化飞书文档/电子表格，读取 `spec/feishu-document-writing-special-flow-spec.md`，按文本可见性、行高、合并、样式和导出复查流程执行。

遇到不确定事项必须先确认；禁止主观改动设计、命名规则、模板字段或对外公告口径。

## 触发场景

当用户提供飞书文档、知识库、电子表格或本地方案内容，并要求从策划案、详细方案、功能说明、交互说明中整理公告需求或版本公告文案时使用本 Skill。

典型请求：

- “读这个详细方案，生成版本更新公告。”
- “根据策划案写公告需求。”
- “把这次功能优化提炼成公告内容。”
- “生成 ROK 公告文案并写入版本公告内容表。”

## 总体流程

1. 解析输入链接并确定对象类型。
2. 确认项目：必须是 `rok`、`cod`、`beagle` 之一；用户未说明时先询问。
3. 拉取正文或读取相关页签，定位详细方案和公告可感知内容。
4. 若项目为 `rok`，读取 ROK 文案术语库校准术语；`cod`、`beagle` 暂按用户提供术语或原文生成，并标注术语库待接入。
5. 提炼玩家可感知更新，过滤内部实现、开发排期、埋点、配置字段和未公开限制。
6. 对照 `reference.md` 判断公告类型和写法形态：宣传型公告、规则说明型公告、优化说明型公告、限制/补偿型公告；不要把规则说明写成活动宣传。
7. 若项目为 `rok`，对照 `reference.md` 的 `ROK 润色列沉淀` 校准短版公告语气；必要时读取 ROK 版本公告内容表历史页签的 `润色（文案）` / `润色` 列做近似案例比对。
8. 按公告类型和更新规模选择文案形态，生成可直接审阅的公告草稿。
9. 如需写入飞书文档、页签或 ROK 版本公告内容表，先确认用户意图；未确认时只输出草稿。

## 项目确认

生成前必须确认项目：

| 项目 | 处理方式 |
| --- | --- |
| `rok` | 读取 ROK 文案术语库；如用户要求，可写入 ROK 版本公告内容表 |
| `cod` | 当前术语库待接入；使用策划案原文和用户提供术语，输出中标注待术语复核 |
| `beagle` | 当前术语库待接入；可参考仓库已有 Beagle 术语资料，但正式公告用词仍需用户确认 |

如果用户没有说明项目，必须先询问，不要默认使用 ROK。若用户提供多个项目或跨项目复用方案，逐个项目生成，不混用术语。

## 链接解析

| URL 类型 | 处理方式 |
| --- | --- |
| `/docx/<token>` | token 可直接用于 `docs +fetch` / `docs +update` |
| `/doc/<token>` | token 可直接用于 `docs +fetch` / `docs +update` |
| `/wiki/<token>` | 先解析 wiki node，获取真实 `obj_type` 与 `obj_token` |
| `/sheets/<token>` | 使用 `sheets +info` 定位相关页签，再按范围读取 |

Wiki 链接必须先查真实对象：

```bash
lark-cli wiki spaces get_node --params '{"token":"<wiki_token>"}' --as user
```

根据 `obj_type` 切换到 `docs` 或 `sheets` 流程。不要把 wiki token 直接当作文档 token。

## 拉取内容

云文档优先使用：

```bash
lark-cli docs +fetch --doc "<doc_url_or_token>" --as user
```

大文档分页拉取：

```bash
lark-cli docs +fetch --doc "<doc_url_or_token>" --offset 0 --limit 50 --as user
lark-cli docs +fetch --doc "<doc_url_or_token>" --offset 50 --limit 50 --as user
```

电子表格先读取页签信息：

```bash
lark-cli sheets +info --url "<sheet_url>" --as user
lark-cli sheets +read --url "<sheet_url>" --range "<sheet_id>!A1:Z120" --as user
```

如果读取结果大量为空，但原表可能有截图、合并单元格、附件或画板，不能视为没有内容；应继续获取关键图片或向用户确认。

## ROK 术语与公告表

ROK 文案术语库：

```text
https://t6bdpf8yjg.feishu.cn/sheets/shtcniOIjRR4rkTXeBONDSKwATc
```

使用规则：

- 生成前用 `sheets +info` 确认术语库页签，再读取相关范围。
- 优先校准游戏名词、活动名、系统名、道具名、玩家称谓、玩法专有名词。
- 术语库没有覆盖的名词保留策划案原文，并在 `待确认事项` 中提示文案复核。
- 不把整张术语库硬编码进 Skill；每次执行按需读取。

ROK 版本公告内容表：

```text
https://t6bdpf8yjg.feishu.cn/sheets/shtcnueF0Ay6wwuoCFih0ZTTkTe?sheet=RTNVGw
```

读取历史润色时：

- 不按固定列号硬读。先读取表头，优先找 `润色（文案）`，其次找 `润色`；早期参考页签可能把正式公告文本放在 `备注/解释/截图` 列，需要结合表头和内容判断。
- `润色（AI）` 只能作为对比参考，不作为最终文案口径；若 `润色（文案）` 与 `润色（AI）` 冲突，优先采用文案列的短版、准确和保守表达。
- 润色列常包含文案同学 @ 信息，提炼样式时忽略 @ 对应人，只读取实际公告正文。
- 历史页签用于学习语气、结构和句式，不作为新需求事实来源；新公告事实仍以当前策划案/详细方案为准。

仅在用户明确要求“写入公告表 / 同步到版本公告内容表 / 填到公告内容表”时执行。写入前必须用 `sheets +info` 和表头读取确认真实列名、列位置和权限，不按固定列号硬写。

字段映射：

| 字段 | 填写规则 |
| --- | --- |
| `填写人` | 当前操作者；如无法自动识别，写 `待确认` 或先询问用户 |
| `公告内容` | 用户确认后的最终公告文案，不写内部分析、提炼依据或待确认事项 |
| `备注解释截图` | 从详细方案选择一张最能代表整个方案的图；无法直接写图时保留来源页签、行号/单元格和画面摘要，并提示需人工补图 |
| `识别组别` | 默认 `系统组`，除非用户另行指定 |

写入完成后读取目标行复查，确保公告内容、填写人和识别组别可读；截图如果无法通过读取验证，需要在回复中说明当前状态。

## 代表截图选择

从 `详细方案` 中选择最能代表整个方案内容的一张图，优先级如下：

1. 核心入口或主界面。
2. 玩法流程总览、活动主视觉或主活动页。
3. 关键规则界面、奖励界面或结算界面。
4. 若是赛季或大系统，优先选能体现主题、核心玩法或新系统全貌的图。

不要默认使用美术包装图、纯资源图、无功能信息的视觉稿。若详细方案没有可用图，写 `待策划补充代表截图`。

## 公告提炼规则

先提炼“玩家可感知更新”，再组织文案。不要把以下内容写进公告：

- 后台实现、接口、配置字段、埋点、日志、数据表。
- 内部代号、研发阶段、需求拆分、负责人、排期。
- 未确认的开放时间、奖励数量、灰度范围、商业口径。
- 玩家不可感知的重构、兼容、修 bug 细节。

活动类公告必须先读背景和规则细节，再写文案：

- 背景：读取 `设计目的`、`背景`、`玩家反馈`、`线上数据`、`讨论`，用玩家视角说明为什么新增或优化。
- 参与对象：谁能参加，例如全体执政官、指定赛季、指定王国、回归玩家、国王或联盟成员。
- 开放节点和持续时间：版本更新后、赛季阶段、K 前夕、活动持续天数、BP 持续天数、商店开放期。
- 替代关系：如果新增活动会取消、合并或替代旧活动，公告正文必须优先写清“新增了什么”和“取消/替代了什么”，避免玩家误以为旧活动缺失。例如新增失落之地活动“中期结算”时，应写明其在所有征服赛季中期开启，同时取消原有“清剿敌军”活动。
- 参与方式和次数：抽取、购买、兑换、任务、投票、竞猜等；每日次数、总次数、购买上限、折扣次数必须写清。
- 奖励和商店：可获得或兑换的关键道具、货币、商店层级、解锁条件、动态上限。
- 限制和回收：跨赛季、移民、地区、赛季差异、货币回收等；标注待定的内容不要写进正式公告。

不能用“更多奖励”“快速成长”“关键资源”等泛化词替代明确数值。若关键参与条件、次数、上限或奖励未确认，必须写入 `待确认事项`。

版本公告中的新增活动若只是单条更新，不要把详细规则全部展开成规则说明页；优先压缩为：活动场景/名称 -> 开放范围或节点 -> 替代旧活动 -> 核心参与行为 -> 奖励结算方式。

规则说明型公告必须单独处理：

- 识别信号：`可购买次数`、`购买上限`、`次数计算`、`注册时间`、`进入征服赛季时间`、`完整周期`、`不计入`、`共享上限`、`无可购买次数`、`无法继续参与`、`跨赛季移民` 等。
- 先确认核心规则对象：本次是在说明哪个规则，而不是整个活动。例如“装备冲刺可购买次数计算”，不是“成长追赶活动上线”。
- 必须提炼关键定义：时间点、周期、上线时间、注册服、进入征服赛季时间、图纸加入时间等。
- 必须保留条件分支：用“若……则……”拆开说明，不能压缩成一句泛化描述。
- 必须保留计算规则：周期长度、每周期次数、购买上限、共享上限、不足周期不计入等。
- 必须保留负向规则：无次数、不能参与、跨赛季后活动结束、移民到未开启王国无法继续参与等。
- 输出顺序优先为：背景原因 -> 关键定义 -> 计算规则 -> 道具/奖励差异 -> 限制例外。

按更新规模选择文案形态。详细判断与示例见 [reference.md](reference.md)。

## 输出格式

默认输出：

```markdown
# 公告需求草稿

## 项目信息

| 项目 | 内容 |
| --- | --- |
| 项目 | <rok / cod / beagle> |
| 来源 | <策划案标题 / 页签 / 链接> |
| 术语库状态 | <已读取 ROK 术语库 / 术语库待接入 / 待用户提供术语> |

## 更新规模判断

<小优化 / 较大系统优化 / 大系统上线 / 赛季更新>：<一句话判断依据>

## 公告文案

<可直接复制审阅的玩家公告文本>

## 提炼依据

| 序号 | 更新点 | 来源 |
| --- | --- | --- |
| 1 | <从详细方案提炼的玩家可感知内容> | <章节/页签/行号或图片来源> |

## 待确认事项

| 序号 | 问题 |
| --- | --- |
| 1 | <只保留影响公告发布口径的问题> |
```

如果没有待确认事项，不输出空表，改写为：`本次暂无影响公告口径的待确认事项。`

## 写回规则

- 写入或修改飞书文档/表格前必须确认用户意图。
- 不覆盖原策划案正文；只能追加、插入或替换用户明确指定的公告需求区块。
- 若已有 `公告需求` 页签或区块，先确认追加、替换还是保留旧版。
- ROK 版本公告内容表只写最终公告内容和指定字段，不写中间分析。

## 完成检查

完成输出或写回前检查：

- 已确认项目为 `rok`、`cod` 或 `beagle`。
- 已读取本目录 `reference.md`，并按其中的更新规模、ROK 润色规则和输出模板选择写法。
- 已优先读取详细方案和公告可感知内容。
- ROK 项目已读取术语库或说明术语库读取失败原因。
- 文案没有内部实现、未确认时间、配置字段和过度承诺。
- 如写入 ROK 公告内容表，已确认字段映射、代表截图和 `识别组别=系统组`。
