# 策划案阅读与音频需求

## 必读前置

在执行本 Skill 前，先读取并遵守：

1. `rules/rules.md`：项目规则、回复前缀、确认边界、会话记录要求。
2. `mistakes/`：先检索相关错题，尤其是图片读取、输出格式、中间交付来源误读。
3. `C:/Users/liuweichen/.agents/skills/lark-shared/SKILL.md`：身份、权限、scope、安全规则。
4. `C:/Users/liuweichen/.agents/skills/lark-doc/SKILL.md`：飞书文档 URL/token 处理、`docs +fetch`、`docs +update`、媒体下载限制。
5. 如遇电子表格或相关页签，读取 `C:/Users/liuweichen/.agents/skills/lark-sheets/SKILL.md`。
6. 如需写入、创建或格式化飞书文档/电子表格，读取 `spec/feishu-document-writing-special-flow-spec.md`，按其中的文本换行、行高、合并、样式和导出复查流程执行。

遇到不确定事项必须先确认；禁止主观改动设计、命名规则、模板字段或产出格式。

## 触发场景

当用户提供飞书文档、知识库或表格链接，并要求从策划案、详细方案、功能说明、交互说明或相关页签中整理音频需求时使用本 Skill。

典型请求：

- “读这个详细方案，整理音频需求。”
- “根据飞书文档补一个音频需求页签。”
- “整理音乐需求和音效需求，参考音频需求模板。”

## 总体流程

1. 解析飞书链接并确定对象类型。
2. 拉取正文或读取相关页签。
3. 定位详细方案、功能说明、交互流程、页面说明、活动规则等功能口径内容。
4. 判断是否存在新场景或明显不同的关键场景屏；新场景进入时默认要考虑是否需要独立音乐循环。
5. 按功能模块整理 `音乐需求` 和 `音效需求`；音效需求只要存在点击、交互、动画或反馈，就要拆出让策划补充。
6. 每个音乐/音效细分需求项都必须增加 `对应切图` 字段，优先引用详细方案中的已有截图、流程图、界面图或表格内嵌图片，避免音频同学只能看到纯文本描述而无法对应画面。
7. 如果方案中出现对话、台词、角色语音、NPC 提示、机器人提示等内容，单独新增 `配音需求`。
8. 写回飞书文档或表格前必须确认用户意图；若用户仅要求整理，先输出草稿。
9. 如果用户要求写回飞书并且存在音频需求登记 Base，完成音频需求页签后必须先进行二次确认，询问用户是否需要添加到需求管理表；只有用户确认后，才在登记表中新增对应记录：指向 `Beagle 7 日版本系统项 音频需求`，填写策划案链接、需求任务名和对应操作者策划名字；除非用户另有要求，其他字段保持空白。

## 链接解析

| URL 类型 | 处理方式 |
| --- | --- |
| `/docx/<token>` | token 可直接用于 `docs +fetch` / `docs +update` |
| `/doc/<token>` | token 可直接用于 `docs +fetch` / `docs +update` |
| `/wiki/<token>` | 先解析 wiki node，获取真实 `obj_type` 与 `obj_token` |
| `/sheets/<token>` | 使用 `sheets +info` 定位相关页签，再按范围读取 |

Wiki 链接必须先查真实对象：

```bash
lark-cli wiki spaces get_node --params '{"token":"<wiki_token>"}' --as user
```

根据 `obj_type` 切换到 `docs` 或 `sheets` 流程。不要把 wiki token 直接当作文档 token。

## 拉取内容

云文档优先使用：

```bash
lark-cli docs +fetch --doc "<doc_url_or_token>" --as user
```

大文档分页拉取：

```bash
lark-cli docs +fetch --doc "<doc_url_or_token>" --offset 0 --limit 50 --as user
lark-cli docs +fetch --doc "<doc_url_or_token>" --offset 50 --limit 50 --as user
```

电子表格先读取页签信息：

```bash
lark-cli sheets +info --url "<sheet_url>" --as user
lark-cli sheets +read --url "<sheet_url>" --range "<sheet_id>!A1:Z120" --as user
```

如果读取结果大量为空，但原表可能有截图、合并单元格、附件或画板，不能视为没有内容；应继续获取关键图片或向用户确认。

## 图片与截图阅读要求

如果返回内容包含 `<image>`、`<file>`、`<whiteboard>`，或飞书表格/文档中明显存在截图、长图、流程图、界面示意：

- 先判断图片是否承载入口、页面状态、流程、弹窗、成功/失败反馈或场景切换。
- 核心方案在图片中时，必须用 `docs +media-download`、表格导出、图片下载或用户附件取得图片后再分析。
- 读图时至少提炼：场景屏名称、进入方式、触发条件、前置条件、操作者、情绪氛围、是否需要独立音乐。
- 图片无法稳定识别时，必须标 `待确认` 或请求高清局部图，不要补脑。
- 图片只体现视觉包装、图标、素材占位时，不因此新增音乐需求。

## 对应切图要求

音乐需求和音效需求不能只给纯文本描述。每个细分需求项都必须增加 `对应切图` 字段，用来帮助音频同学把需求和画面对应起来。

切图来源规则：

1. 默认只能使用 `详细方案` 页签/章节中的流程图、界面截图、长图、表格内嵌图片。
2. 只有当策划明确特殊说明，或单独提供了音频需求可用图片时，才可以使用详细方案以外的 UI/美术/包装/资源页签图片。
3. 若详细方案没有可用截图，且策划没有额外指定图片来源，不要擅自改用其他页签图片；应写 `待策划补充流程假图/录屏`，并在待确认事项中说明缺图风险。
4. 若图片实体无法通过 API 稳定复制，至少写明来源页签、行号或单元格位置和画面内容摘要，例如：`详细方案 R34：大地图操作屏增加表情按钮 / 点击呼出表情`。

使用规则：

- `对应切图` 只服务于定位画面，不替代 `关键交互`、`操作者`、`前置条件`。
- 如果已经能在音频需求页签中直接显示图片，`对应切图` 行右侧不再保留来源说明文字，只保留字段标签和图片本体；来源说明可在内部脚本或会话记录中保留。
- 切图必须和当前细分需求项直接相关，不要把整份方案首页或无关包装图泛用到所有需求项。
- 如果一个需求项对应多个画面状态，可在 `对应切图` 中按换行列出多个来源。
- 如果当前方案没有可用截图，写 `待策划补充流程假图/录屏`，并在待确认事项中说明缺图风险。

## 音乐需求判断

音乐需求整理“新场景”和“明显不同的关键场景屏”。如果策划案引入城镇、玩法空间、活动空间、副本空间、剧情空间等新场景，进入该场景时默认要提出“是否需要新的音乐循环”的需求，除非文档明确沿用现有 BGM。

如果没有新场景，也没有明显不同的关键场景屏，才明确输出“本次未识别到需要新增独立音乐的关键场景屏”。不要因为没有活动包装就忽略新场景音乐循环。

适合提出音乐需求的情况：

- 新增场景或独立空间：城镇、营地、矿区、副本、活动地图、特殊玩法空间等。
- 战斗/副本相关：例如埃及之战、5v5、海战、ARPG 附属玩法等，需要考虑是否使用独立战斗或副本音乐。
- 活动/节日强相关：例如周年庆、特定节日活动界面的专属 BGM。
- 叙事/剧情强相关：多情绪向的剧情对话、故事背景铺垫，需要考虑剧情氛围音乐。
- 视频演示类：CG 动画、PV 宣传片、Splash（启动闪屏）等，需要考虑演示配乐或开屏音乐。
- 不确定的全屏系统界面：如果是新的全屏大系统，且不确定是否需要替换主城 BGM，必须标记为 `待音频同学评估`，不要直接判定为不需要音乐。
- 活动主界面、IAP 界面、节日/周年庆/主题活动界面等风格化页面。
- 特殊 loading、入口屏、剧情开场、章节大图等有独立氛围的页面。
- 与主游戏常规 BGM 明显不同的玩法、探索、危机、胜利、失败、结算页面。
- 文档明确要求新音乐、特殊氛围、情绪转折或独立场景屏。

默认不提出音乐需求的情况：

- 普通按钮点击、弹窗开关、列表刷新、奖励领取等通用 UI 反馈。
- 只属于通用音效或已有基础 UI 音效的动作。
- 只有美术资源描述，没有进入方式、触发条件或场景用途。
- 文档没有体现独立氛围变化，且只是沿用当前系统界面或现有 BGM。

## 音乐情绪

优先从以下选项选择，可多选：

- `紧张压迫`
- `探索神秘`
- `胜利凯旋`
- `悲伤叙事`
- `节日氛围`
- `轻松氛围`

如文档明显表达其他情绪，可新增 `其他：<说明>`。不要为了覆盖选项而强行勾选。

## 音效需求规则

音效需求按功能模块拆分，不替策划细拆具体音效资源，但要把“需要策划补充的交互流程”列出来。只要功能中存在点击、按钮、弹窗、滑杆、选择、派遣、确认、取消、动画、传送、成功/失败反馈等交互或表现，就应提出对应音效需求块。

适合提出音效需求的情况：

- 页面 UI 交互：各类按钮点击、弹窗出现/关闭、奖励结算等。这是最普遍的音效需求，绝大部分新系统都会涉及。
- 场景氛围音（Ambience）：大地图/3D 场景中的奇观、传送门、特殊地貌环境；活动界面内景中的下雪声、壁炉燃烧白噪音等。
- 动画与特效强相关（动作/反馈）：主角/NPC 行为逻辑（如脚步声）、技能释放、Boss 攻击、受击反馈等。
- 视频音效：CG、PV 等演出的配套音效设计。

默认不单独拆音效需求的情况：

- 普通报错提示、发送冷却、内容不存在、达到上限等轻量 UI 错误反馈，通常沿用通用错误提示音或 UI 反馈，不单独作为音效需求块。
- 如果错误反馈带有强演出、特殊动画、失败结算、战斗失败或玩法核心情绪变化，再合并到对应功能交互块中说明，不单独拆成“报错音效需求”。

复合多个需求的策划案必须按功能拆分，例如：

- 导航功能交互
- 代驾功能交互
- 城镇矿交互
- 打包/迁入/展开交互

每个音效需求块只写功能、对应切图、关键交互、操作者和前置条件，不自行指定具体音效名、音色或资源方案。

如文档已明确操作者或前置条件，必须补在对应功能块下；不明确时写 `待策划补充`。

## 配音需求规则

只要角色或系统有明确的文案、台词内容，且需要发声讲话，就包含配音需求。当方案里出现对话需求时，音频需求中必须单独增加 `配音需求` part。对话需求包括但不限于：

- 角色、NPC、机器人、小助手、小动物等说出的对白。
- 弹窗或引导中明显应由角色发声的提示文案。
- 新手任务、剧情、功能引导、代驾提示、战斗喊话等带说话人的文本。
- 文档中出现 `对话`、`台词`、`语音`、`旁白`、`说话人`、`机器人提示` 等关键词。

以下情况不默认输出配音需求：

- 普通 UI 文案、按钮文案、标题、说明、错误提示、气泡窗文案，仅作为界面文字展示时，不按配音需求处理。
- IM 邀请、组队申请、系统通知等气泡窗文案，即使语气上像一句话，也不默认生成配音需求；除非文档明确要求语音播报、角色/系统发声或带有明确说话人。

配音需求只罗列策划案已出现或可从明确文本提取的对话，不编写新台词。每条配音需求必须包含：

- `对话内容`：原文案或待补充占位。
- `说话人`：如机器人、NPC、系统旁白、具体角色名；不明确写 `待策划确认`。
- `情绪`：如热情引导、轻松提示、紧张警告、疑惑、鼓励、失败提醒；不明确写 `待策划确认`。
- `场景触发`：说明在什么界面、流程节点、条件下触发。

如果方案没有任何对话或带说话人的提示，不输出 `配音需求` part。

### 固定样式规范

以下样式已从参考页签导出为 xlsx 后解析，后续写入飞书表格时优先直接套用，不必每次重新读取参考表：

| 层级 | 布局 | 字体 | 底色 | 字色 | 对齐 |
| --- | --- | --- | --- | --- | --- |
| 总标题 | `A1`，可横向合并至内容宽度 | Calibri 36，加粗；后续批量重设样式时必须单独恢复大标题字号 | `#FAC603` | `#000000` | 垂直居中 |
| 一级分区序号 | 分区行左侧数字格，如 `B3`、`B21` | Calibri 12 | `#3A3E43` | `#FFFFFF` | 水平/垂直居中 |
| 一级分区标题 | 分区行标题格，如 `C3`、`C21` | Calibri 12，加粗 | `#FAC603` | `#000000` | 垂直居中 |
| 条目序号 | 条目行数字格，如 `C5`、`C23` | Calibri 10，加粗 | `#FAC603` | `#000000` | 水平/垂直居中 |
| 条目标题 | 条目行标题格，如 `D5`、`D23` | Calibri 10，加粗 | `#FAF1D1` | `#000000` | 垂直居中 |
| 字段标签 | `C` 列，如 `C25` 的 `一句话解释：`、`C27` 的 `音效类型`、`C29` 的 `音乐情绪`、`C41` 的 `操作者：`、`C43` 的 `前置条件：` 等；不要放在 `B` 列 | Calibri 10，加粗 | 无填充 | `#000000` 或 `#1F2329` | 垂直居中 |
| 正文内容 | 字段标签右侧内容格 | Calibri 10 | 无填充 | `#000000` | 垂直居中 |

布局规则：

- 保留参考表的大量留白：标题、分区、条目和字段行之间至少空 1 行。
- `B` 列只用于一级分区序号和参考模板中的固定占位；字段标签/解释文本放在 `C` 列，正文内容从 `D` 列开始。
- 凡是底色为 `#3A3E43` 的单元格，字体必须为白色 `#FFFFFF`，不能被全局字体样式覆盖成黑色。
- 批量设置全表默认字号后，必须再次单独恢复 `A1` 总标题为 Calibri 36 加粗，避免大标题被压回正文字号。
- 长文本不要只依赖表格自动换行；对 `说明：`、`一句话解释：`、`关键交互：`、`前置条件：`、`待确认事项` 等超过一行的内容，写入时主动插入换行，并按最终实际行号设置足够行高。
- 行高不能用上一版固定行号生搬硬套；内容块数量变化后，必须根据最终行号重新计算并设置长文本行、分区行、条目标题行和空白行高度。
- 飞书表格没有确认可用的真正自动适应行高 API；写入完成后必须执行一次“模拟 autofit”：读取最终所有行，按显式换行数和文本长度计算每行 `fixedSize`，逐行设置行高，再导出 xlsx 复查。
- 黄色系以主色 `#FAC603` 为基准；条目标题使用浅黄色变体 `#FAF1D1`，用于形成同一黄色体系下的层级变化。
- 音乐需求按块输出：条目标题下依次放 `一句话解释：`、`对应切图：`、`音效类型`、`音乐情绪`。
- 音乐情绪按纵向选项展示，可在右侧用 `√` 标记选中项；也可在内容格直接写多选情绪，但优先保持参考表的纵向选项感。
- 音效需求按功能块输出：条目标题下依次放 `对应切图：`、`关键交互：`、`操作者：`、`前置条件：`；有图时不要再额外输出 `策划补充提示`。
- 配音需求使用表头行：`序号 / 对话内容 / 说话人 / 情绪 / 场景触发`；表头可使用字段标签样式。
- 参考页签没有显式边框；如通过 API 写入后可读性不足，可加浅灰边框 `#B7B7B7` 作为降级样式，但不要让边框抢过黄色分区层级。

写入飞书电子表格时，优先按上方固定样式规范创建新 `音频需求` 页签并设置样式。若当前 CLI/API 无法设置样式，只输出同结构内容，并说明“样式需在飞书中按参考音频需求页签套用”。不要臆造未记录的颜色值、字体参数或样式 API 返回。

## 音频需求登记 Base

当项目存在统一音频需求登记 Base 时，生成并写回音频需求后先询问用户是否需要同步到需求管理表。只有用户二次确认需要后，才新增登记记录。

默认登记规则：

- 目标表：用户指定的音频需求登记表；当前 7 日版本约定为 `Beagle 7 日版本系统项 音频需求`。
- 必填内容：策划案链接、需求任务名、对应操作者策划名字。
- 其他字段：默认不填，除非用户明确要求补充。
- 二次确认：完成生成后必须明确询问“是否需要添加到需求管理表中”；用户未确认前不写入 Base。
- 操作前必须先解析 wiki/base 链接、读取表结构和字段类型，只写可写字段；不要向公式、查找引用、创建时间、更新时间等只读字段写值。
- 如果缺少 Base 读写权限，先按最小权限发起 `lark-cli auth login`，不要改用未验证的接口绕过权限。

当前 7 日版本登记表信息：

- wiki 链接：`https://t6bdpf8yjg.feishu.cn/wiki/LOC3wnhiPi68T1kNZm7cJIeVn7e?table=tblfT7sBWJXgRqXY&view=vewkFfkufW`
- 解析后的 Base token：`AZHgbHFazaDLLBs50cJcoJhKnMf`
- 表 ID：`tblfT7sBWJXgRqXY`
- 视图 ID：`vewkFfkufW`
- 表名：`Beagle 7 日版本系统项 音频需求`
- 常用字段：
  - `策划案链接`：文本 URL 字段，填音频需求页签或策划案链接。
  - `需求/任务名`：文本字段，填如 `小队音频需求`。
  - `DRI负责人`：人员字段，填对应操作者策划；写入时必须使用人员 open_id，例如 `[{ "id": "ou_xxx" }]`。

登记流程：

1. 生成并写回 `音频需求` 页签后，先二次确认：询问用户是否需要添加到需求管理表中。
2. 用户确认需要后，再执行 Base 登记；用户未确认或表示不需要时，到此停止，不新增记录。
3. 如果用户给的是 `/wiki/<token>` 链接，先执行 `lark-cli wiki spaces get_node` 获取真实 `obj_type` 与 `obj_token`；`obj_type=bitable` 时把 `obj_token` 作为 `--base-token`。
4. 用 `lark-cli base +table-list --base-token <base_token> --as user` 确认目标表名与表 ID。
5. 用 `lark-cli base +field-list --base-token <base_token> --table-id <table_id> --as user` 读取字段结构，只选择可写字段。
6. 如果要写人员字段，先确认策划名字对应的 open_id；可从策划案内 `@人` mention 的 `token/open_id`、通讯录搜索或用户明确提供的信息获取。不要把纯姓名字符串写入人员字段。
7. 用 `lark-cli base +record-upsert` 新增记录，只写用户要求字段。当前默认只写 `策划案链接`、`需求/任务名`、`DRI负责人`，其他字段不传。
8. 返回创建成功的 `record_id`，并把本次登记写入 `session/session.md`。

权限最小集：

- 解析 wiki：`wiki:node:read`
- 列表：`base:table:read`
- 字段：`base:field:read`
- 新增记录：`base:record:create`；如后续更新已有记录，还需要 `base:record:update`

Windows/PowerShell 写入注意：

- 中文字段名的 JSON 不要直接用 PowerShell 字符串硬传，容易出现 `--json invalid JSON object` 或中文乱码。
- 推荐用 Python 构造 `json.dumps(payload, ensure_ascii=True)` 后调用 `lark-cli.cmd base +record-upsert`。
- 示例：

```python
import json
import subprocess

payload = {
    "策划案链接": "<音频需求页签链接>",
    "需求/任务名": "<系统名>音频需求",
    "DRI负责人": [{"id": "ou_xxx"}],
}

subprocess.run([
    "lark-cli.cmd", "base", "+record-upsert",
    "--base-token", "<base_token>",
    "--table-id", "<table_id>",
    "--json", json.dumps(payload, ensure_ascii=True),
    "--as", "user",
], check=True)
```
