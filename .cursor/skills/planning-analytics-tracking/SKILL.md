# 策划案阅读与数据打点

## 必读前置

在执行本 Skill 前，先读取并遵守：

1. `C:\Users\liuweichen\.agents\skills\lark-shared\SKILL.md`：身份、权限、scope、安全规则。
2. `C:\Users\liuweichen\.agents\skills\lark-doc\SKILL.md`：飞书文档 URL/token 处理、`docs +fetch`、`docs +update`、媒体下载限制。
3. 项目规则：`rules/rules.md`。遇到不确定事项必须先确认，禁止主观改动设计、命名规则或产出格式。
4. 错题清单：`mistakes/`。如果策划案主要是长图、截图、画板或 OCR 不稳定内容，先标记 `待确认`，不要补脑。
5. 如需写入、追加、替换或格式化飞书文档/电子表格，读取 `spec/feishu-document-writing-special-flow-spec.md`，按其中的文本可见性、行高、合并、样式和导出复查流程执行。

## 触发场景

当用户提供飞书文档链接，并要求从策划案、详细方案、功能说明、交互说明中整理数据打点或埋点需求时使用本 Skill。

典型请求：

- “阅读这个策划案，整理数据打点。”
- “看飞书文档里的详细方案，新增一个数据打点页签。”
- “根据这个功能开发内容，给出埋点需求和字段表。”

## 总体流程

1. 解析飞书链接。
2. 拉取文档正文。
3. 定位「详细方案」和功能相关章节。
4. 先提炼本功能要回答的数据问题，再推导事件表与字段。
5. 优先尝试写入原生数据打点页签；不支持时回退为文档内 `## 数据打点` 区块。
6. 在写回前向用户确认最终打点内容，除非用户已明确要求直接写入。

## 链接解析

根据 URL 类型选择处理方式：

| URL 类型 | 处理方式 |
| --- | --- |
| `/docx/<token>` | token 可直接用于 `docs +fetch` / `docs +update` |
| `/doc/<token>` | token 可直接用于 `docs +fetch` / `docs +update` |
| `/wiki/<token>` | 先解析 wiki node，获取真实 `obj_type` 与 `obj_token` |
| `/sheets/<token>` | 这是电子表格，不按策划文档处理；询问用户是否要读取表格内容 |

Wiki 链接必须先查真实对象：

```bash
lark-cli wiki spaces get_node --params '{"token":"<wiki_token>"}' --as user
```

只在 `obj_type` 为 `docx` 或 `doc` 时继续。若是 `sheet`、`bitable`、`file`、`slides` 等，先告知用户当前 Skill 主要处理云文档，并询问是否切换到对应 Skill。

## 拉取正文

优先使用：

```bash
lark-cli docs +fetch --doc "<doc_url_or_token>" --as user
```

大文档分页拉取：

```bash
lark-cli docs +fetch --doc "<doc_url_or_token>" --offset 0 --limit 50 --as user
lark-cli docs +fetch --doc "<doc_url_or_token>" --offset 50 --limit 50 --as user
```

如果返回 Markdown 中包含 `<image>`、`<file>`、`<whiteboard>`，或飞书电子表格/文档中明显存在截图、长图、界面示意、流程图：

- 先判断图片是否承载功能实现信息，而不是只判断文字正文是否足够。
- 若核心方案在图片、长图、画板、流程图或 UI 截图中，必须使用 `docs +media-download`、表格导出、图片下载或用户提供附件等方式取得图片，再读图分析。
- 读图时至少提炼：入口/按钮、页面状态、流程节点、成功/失败反馈、弹窗/提示、限制条件、奖励/消耗展示。
- 生成打点前，先在内部对照“文字说明 + 图片信息”合并判断；不能只按文本中的标题、备注或零散说明生成字段。
- 如果图片无法下载、分辨率不足、OCR/视觉识别不稳定，必须先二次询问或请求用户提供高清局部图；不要继续生成最终打点需求。
- 对图片中只属于 `美术需求`、视觉包装、动效表现、素材占位的内容，仍按黑名单处理，不作为功能实现口径。

### 图片与截图阅读要求

详细方案中的图片常常承载真实交互和状态信息，尤其是飞书电子表格里的内嵌截图。执行时必须遵守：

- **先读图再定表**：如果图中包含流程、界面按钮、操作反馈或状态变化，先理解这些内容，再决定事件表和字段。
- **图文互证**：文字和图片冲突时，不直接下结论；优先标 `待确认` 或二次询问。
- **不凭空扩展**：图片只出现视觉样式、布局、图标资源时，不因此新增打点字段。
- **不忽略空白表格**：如果 `sheets +read` 读到大量空值，但原表看起来有内容，优先怀疑内容在图片、合并单元格、附件或画板中，不能视为“无需求”。
- **输出可追溯**：在 `待确认事项` 或总结中说明哪些字段来自图片理解，哪些需要策划/数据确认。

## 阅读范围

优先读取以下标题或关键词所在章节：

- `详细方案`
- `功能说明`
- `玩法说明`
- `交互说明`
- `流程`
- `状态`
- `规则`
- `结算`
- `奖励`
- `失败`
- `异常`
- `接口`
- `数据`
- `埋点`
- `日志`

如果找不到「详细方案」，读取与功能实现最相关的章节，并在输出开头说明来源章节。不要基于标题以外的猜测扩展功能范围。

### 读取黑名单

以下页签、章节或标题内容**不得作为功能实现口径读取**，也不得据此生成埋点字段：

- `美术需求`
- `美术`
- `视觉需求`
- `表现需求`
- `资源需求`
- `动效需求`
- `音频需求`
- `素材需求`

黑名单原因：这些内容通常是中间交付、资产/表现/包装说明，不代表最终功能实现逻辑，容易把视觉资源、表现占位、临时方案误读成功能字段。

执行规则：

- 工作簿中存在标题含 `美术需求` 的页签时，默认跳过，例如 `小队-副本界面美术需求`。
- 若功能页签中引用美术需求，只把它当作上下游说明，不进入埋点推导。
- 只有当用户明确要求“分析美术需求的打点/资源埋点/表现埋点”时，才读取黑名单内容，并在输出中注明这是美术/表现口径，不是功能实现口径。
- 如果某字段只来自美术需求、视觉表现或资源说明，默认删除；不确定时先二次询问。

## 不确定环节二次询问

遇到会影响产出结构或功能口径的不确定事项时，必须先二次询问，不要只在 `字段备注` 或 `待确认事项` 里带过。优先使用 `AskQuestion`，一次只问 1-2 个关键问题。

必须询问的情况：

- **范围不确定**：文档含多个相关页签，无法判断是否都要纳入打点，例如主功能、战斗预期、奖励系统、社交系统分散在不同页签。
- **功能归属不确定**：某功能属于其他团队/其他策划案，如“战斗数据统计见战斗组策划案”，无法判断是否应在当前页签设计字段。
- **来源落入黑名单**：字段或逻辑只在 `美术需求`、`视觉需求`、`音频需求` 等黑名单页签中出现，无法确认是否为最终功能实现。
- **图片承载核心方案但无法读取**：详细方案里的截图、长图、流程图或画板包含核心交互/状态/流程，但当前工具无法稳定识别。
- **表名口径不确定**：同一行为可归入多个模块名，或团队已有命名前缀/登记表但当前文档未说明。
- **公共维度边界不确定**：字段可能由登录、用户维表、SDK 公共头、战斗专表提供，无法判断是否应重复写入功能表。
- **关键枚举不确定**：入口来源、结果枚举、难度枚举、奖励类型、队伍人数、匹配阶段等会影响字段备注或字段设计。
- **写回方式不确定**：目标文档已有 `数据打点` 页签/区块，或用户没有说明追加、替换、另建页签。
- **负责人不确定且用户要求正式交付**：文档多人协作但未明确数据打点负责策划。

可以不阻塞、但必须在 `字段备注` 或 `待确认事项` 中标记的情况：

- 配置值未定但不影响字段存在，例如默认 150s、每日上限 X、奖励数量 x。
- 复合字段类型仍写 `string`，后续只需与数据/后台确认字段说明中的字符串格式。
- 留坑/七日不做功能只需记录点击和未开放结果，不需要询问完整闭环。

询问示例：

```text
这个表格同时包含副本界面、匹配规则、战斗预期和友情点系统。请确认本次数据打点是否只覆盖“副本界面开发详细方案”，还是要把匹配、结算、友情点也纳入同一个数据打点页签？
```

```text
文档中战斗数据统计写明“见战斗组策划案”。本次是否只保留结算侧关键字段（评级、耗时、领奖、点赞），不展开完整战斗明细？
```

## 推导规则

### 用户修订后的关键教训

生成数据打点时优先遵守以下规则，这是从用户修订后的 `数据打点` 页签、数据组反馈和埋点设计参考表沉淀出的基准：

- `## 需求描述总览` 的结构不变，仍按 `功能名 / 需求描述 / 目的` 提炼核心复盘问题。
- 不要硬搬“轻量映射表”或“复杂字段块”。每次必须先判断该需求应复用哪个已有承接表、新增哪类事件表、是否只补充枚举维表，再推荐结构。
- 设置项、开关、滑条、多档选择、功能列表、配置枚举、外部 push 类型等内容，只有在已有统一承接表能记录运行时行为时，才输出轻量表：`change_type / *_content / 中文分类 / 备注 / UI类型`。
- 教程步骤、loading 流程、UI 点击、设置变更等看似“枚举”的内容，如果参考设计表已有承接表（如 `lgrolefte`、`lgroleloading`、`lgroleuiclick`、`lgrolesettingchange`），必须先说明承接表、上报端、记录时机和核心字段，再决定是否补充枚举映射表。
- 复杂玩法流程、结算、奖励、战斗、任务进度、资源流动、抓捕、养成等需要记录上下文和结果的数据，使用“表名行 + 功能描述行 + 字段小表”的纵向块结构。
- 不要为每个字段重复写 `负责策划`。负责人放在页签顶部，例如 `负责策划 | xxxx`。
- 不要默认加入“字段归属”列。字段归属可以作为内部判断依据，但最终交付表不展示这一列。
- 不要把所有候选字段都列进去。只保留对当前功能复盘有直接价值的字段；能由公共维度、用户维表、登录表、SDK 自动带出的字段，不放进功能表。
- 同一页面的展示、按钮点击、未开放结果如果分析目标一致，可以合并到一张表，不要机械拆成多张表。
- 对“留坑/七日不做/未开放”的功能，只记录必要的点击、结果、报错，不设计完整闭环字段。
- 对枚举值、版本限制、配置备注，优先写在 `字段备注` 列，而不是塞进字段说明正文。

### 需求提炼边界

本 Skill 产出的是策划侧数据打点需求，不输出数据组后续埋点方案设计字段。

- 输出开头必须先给出“需求描述总览”，按 `功能名 / 需求描述 / 目的` 说明本次打点要回答哪些指标问题。
- 先写清该功能要追踪的核心指标或复盘问题，再决定字段；没有服务明确问题的字段不要默认列为必需。
- 明确来自策划案“数据需求”的字段优先；只是“可能有用/方便分析”的字段必须写成可选或待确认。
- 不在事件表中新增 `记录时机`、`打点侧建议`、`关联说明` 等数据组方案字段，除非用户明确要求。
- 同一系统内相同概念字段保持命名一致。
- 绿色服、特殊版本或其他团队约定前缀如果在文档中出现，表名和字段名按原约定保留；未出现时不要自行添加前缀。
- `待确认事项` 精简到 1 到 3 个关键问题，优先问会影响表结构、功能范围、枚举口径或负责人归属的问题。

### 埋点设计参考表判断流程

如果用户提供或项目已有埋点设计参考表，生成前必须先对照参考表做结构判断，不允许只套模板。当前参考表的关键形态：

- `服务器打点设计` / `客户端打点设计` 是主设计页：先列通用表头，再用 `大类 / 模块 / 表名 / 功能描述` 做承接表目录，后续用 `新增表 / 表名 / 功能描述 / 字段名 / 字段类型 / 字段说明` 做具体字段块。
- `FTE步骤`、`loading步骤`、`settingchange`、`uiclick` 等是枚举/步骤/内容维表：用于补充 `action_type`、`loading_key`、`setting_content`、`ui_type/ui_subtype` 等枚举含义，不替代主事件表。
- `优先级 / 程序对接人 / 程序进度 / 预计完成时间 / 数据组 / 日志平台建表 / QA / 验证结果 / 造数据需求 / QA备注 / 上线状态` 等列属于落地追踪状态，除非用户明确要求，不进入策划侧打点需求。

每次生成前按下面顺序判断：

1. **先判断上报端**：客户端可观测的点击、设置、loading、FTE、卡点、网络心跳优先看客户端承接；服务器权威的战斗结算、资源流动、奖励、任务、订单、抓捕、养成、联盟、邮件优先看服务器承接。
2. **再判断是否已有承接表**：参考表已有类似表名或功能描述时，优先复用并补字段/枚举，不要另造新表。例如教程步骤优先看 `lgrolefte` + `FTE步骤`，设置变更看 `lgrolesettingchange` + `settingchange`，UI 点击看 `lgroleuiclick` + `uiclick`。
3. **再判断记录粒度**：是每次点击、每次设置改变、每个流程步骤、每次结算、每日快照、登录快照、资源进出，还是异常/失败时上报。记录粒度决定表结构，不能只按页面名生成一张映射表。
4. **最后选择明细结构**：已有承接事件表时输出“推荐承接表 + 字段块 + 必要枚举维表”；只有纯配置枚举补充且运行时字段已由统一事件承接时，才只输出轻量映射表。

输出明细前必须先给一个简短的结构判断：

| 判断项 | 推荐 |
| --- | --- |
| 上报端 | 客户端 / 服务器 / 待确认 |
| 承接方式 | 复用 `<已有表>` / 新增 `<候选表>` / 仅补充 `<枚举维表>` |
| 记录粒度 | 每次点击 / 每次步骤变化 / 每次结算 / 每日快照 / 每次资源变化 / 异常时 |
| 明细结构 | 事件字段块 / 枚举映射表 / 事件字段块 + 枚举映射表 |
| 判断理由 | 为什么这样设计，和参考表的对应关系 |

### 事件表拆分

按“业务模块 + 触发时机”拆事件表。常见触发时机包括：

- 入口曝光或打开：`enter` / `show`
- 点击或选择：`click` / `select`
- 开始或提交：`start` / `submit`
- 战斗/玩法结束：`end` / `settle` / `result`
- 失败或异常：`fail` / `error`
- 奖励领取：`reward`

每张表必须能回答一个明确的业务问题，例如：

- 用户是否进入该功能？
- 用户在哪一步流失？
- 用户使用了哪些配置或阵容？
- 结果、耗时、失败原因和奖励是否符合预期？
- 哪些参数影响胜率、评价、转化或留存？

拆表前先问两个问题：

1. 这张表后续会被单独看指标吗？
2. 这些字段是否服务同一个复盘问题？

如果答案是否定的，不要拆表；优先合并到同一功能表或删除字段。

### 表名格式

表名遵循 `reference.md` 中的“参考表名格式”：

- 全小写英文或团队约定缩写。
- 默认不使用下划线；避免中文、空格和临时缩写。
- 推荐结构：`{业务域或玩法}{对象或模块}{生命周期动作}`。
- 示例：小队副本战斗结束可写为 `raidbattleend`。
- 如果团队已有前缀规则（如 `fact_`、`dwd_`、`lc_`），优先沿用。
- 不确定时输出 2 到 3 个候选表名，并标记 `待数据/后台确认`。

### 明细表结构选择

先根据承接表、上报端、记录粒度和复盘问题选择明细表结构，不要默认展开为多张事件字段表，也不要默认压成轻量映射表。

#### 轻量枚举/配置映射表（有条件使用）

适用场景：

- 设置界面、开关、滑条、多档选择、语言选择、画质/音频/操作配置。
- 功能解锁列表、任务类型列表、可配置入口、外部 push 类型、通知类型、社区入口等枚举型内容。
- 数据组已经有统一事件承接运行时字段，例如用户、时间、设备、旧值、新值、结果、来源、版本等。
- 参考表中存在对应枚举维表，例如 `settingchange`、`uiclick`、`FTE步骤`，且本次只是补充枚举项或中文说明。

默认列结构：

| change_type | content_key | 中文分类 | 备注 | UI类型 |
| --- | --- | --- | --- | --- |

列名说明：

- `change_type`：事件或模块归类，例如 `setting_screen`、`setting_audio`、`setting_basic`、`setting_battle`。
- `content_key`：具体设置项、功能项或枚举项 key。可按业务替换为更准确的列名，例如 `setting_content`、`unlock_content`、`push_content`。
- `中文分类`：给策划和分析阅读的中文大类，例如画面设置、音频设置、基础设置、战斗设置。
- `备注`：具体含义说明，例如画面质量、帧率、主音量、屏蔽表情。
- `UI类型`：控件或交互形态，例如多档选择、音量滑动、开关、按钮、外链、敬请期待拦截。

轻量表规则：

- 不重复展开 `old_value`、`new_value`、`default_value`、`apply_result`、`entry_source`、`device_model` 等运行时字段，除非策划案明确要求这些字段由当前功能单独定义。
- 不为了每个设置项拆独立事件表；同类配置尽量放在同一张映射表中。
- `content_key` 优先使用策划案已有 key、配置名或英文枚举；没有英文 key 时再用稳定英文名，不使用临时中文拼音。
- 未开放、七日不做、需要外部系统支持的内容，写在 `备注` 或 `UI类型` 中，例如 `敬请期待拦截`、`七日屏蔽入口`、`服务器支持待确认`。
- 轻量表只能作为“枚举/配置说明”。如果需求需要分析步骤完成、失败、流失、耗时、结算结果、奖励、资源变化，必须同时输出或改用事件字段块。

示例：

| change_type | setting_content | 中文分类 | 备注 | UI类型 |
| --- | --- | --- | --- | --- |
| setting_screen | Graphic | 画面设置 | 画面质量 | 多档选择 |
| setting_screen | FrameRate | 画面设置 | 帧率 | 多档选择 |
| setting_audio | MainVolume | 音频设置 | 主音量 | 音量滑动 |
| setting_basic | HideEmoji | 基础设置 | 屏蔽表情 | 开关 |

#### 复杂事件字段块

适用场景：

- 玩法开始/结束、战斗结算、任务进度、奖励领取、失败异常、社交互动等需要记录业务上下文和结果的数据。
- 单靠枚举映射无法回答核心复盘问题，必须记录对象 ID、配置、结果、进度、奖励、失败原因等字段。
- FTE/教程、loading、UI 点击、设置变更等如果参考表已有承接事件表，也应输出该事件表的关键字段和记录逻辑；枚举表只补充步骤或内容含义。

### 复杂事件字段规则

复杂事件字段表使用五列结构：

| 通用表头 | 字段类型 | 字段说明 | 必要性 | 字段备注 |
| --- | --- | --- | --- | --- |

这里的“通用表头”指字段名，不是跨事件公共字段。

如果写入飞书电子表格，可按用户样例在 `字段说明` 与 `必要性` 之间留一个空列作为视觉间隔，但内容语义仍按上方五列理解。

字段类型只支持 `string`。无论字段语义是数字、枚举、百分比、布尔值、时间戳还是复合结构，最终交付的 `字段类型` 列都填写 `string`。

对 `pet_1`、`player_skill`、战力、耗时、血量百分比等字段，在字段说明或字段备注中写清格式、单位、枚举、取值范围和解析方式，不要把字段类型改成 `int`、`float`、`bool` 或 `json`。

字段说明必须写清：

- 触发时记录的具体含义。
- 枚举值含义。
- ID 来源表（如果策划案出现，如 `RaidBoss表`）。
- 复合字段格式。
- 时间、百分比、伤害、次数等单位。

负责策划：

- 放在整页顶部：`负责策划 | <姓名或待策划确认>`。
- 不要在每一行字段中重复负责人。
- 如果文档中没有负责人，填 `待策划确认`，不要擅自指定。

### 跨表公共维度

如果需要补充 `event_time`、`user_id`、`server_id`、`app_version`、`platform`、`session_id` 等全局字段，不要塞进“通用表头”列的概念里。单独写在表格前：

```markdown
**跨表公共维度（如团队埋点 SDK 已自动采集，可不重复声明）**：
- event_time：事件时间
- user_id：用户 ID
- app_version：客户端版本
- platform：平台
```

## 写回飞书文档

### 原生页签优先

用户已确认优先尝试“飞书原生底部页签”。执行时按以下顺序：

1. 先检查 CLI 是否已有页签能力：
   ```bash
   lark-cli docs --help
   ```
2. 如果没有相关命令，按 `lark-openapi-explorer` Skill 调研 docx OpenAPI，重点查：
   - 是否支持创建原生文档 Tab / Sheet Tab / Page Tab。
   - 是否支持创建块类型来模拟页签。
   - 所需 HTTP 方法、路径、参数、scope。
3. 只有在 API 明确支持且权限满足时，才用 `lark-cli api` 创建原生页签并写入内容。
4. 如果 CLI/API 不支持、权限不足、或官方文档没有明确页签接口，说明限制，并回退为文档内 `## 数据打点` 区块。

不要臆造 docx 页签 API。当前已知 `lark-cli docs` 只有 `+create`、`+fetch`、`+update`、`+search`、媒体与画板相关命令，未提供原生页签命令。

### 章节回退

回退时使用局部追加，不要覆盖全文：

```bash
lark-cli docs +update --doc "<doc_url_or_token>" --mode append --markdown "<markdown>" --as user
```

若用户指定插入位置，可用：

```bash
lark-cli docs +update --doc "<doc_url_or_token>" --mode insert_after --selection-with-ellipsis "<unique_anchor>" --markdown "<markdown>" --as user
```

禁止在无明确确认时使用 `overwrite`。包含图片、画板、文件、评论等不可重建内容的文档尤其不能覆盖。

## 安全与确认

- 写入或修改飞书文档前，必须确认用户意图；若用户已经明确“写入这个文档/新增数据打点页签”，可继续。
- 对不确定的功能含义、字段命名、表名冲突、负责人、字段类型，必须标 `待确认` 或向用户提问。
- 不得改写策划案原有内容，只能新增或替换明确的数据打点区块。
- 若已有 `## 数据打点` 区块，先询问是追加、替换还是保留旧版。

## 验收标准

- 能读取 `/docx/`、`/doc/`、`/wiki/` 链接并正确处理 token。
- 能优先识别「详细方案」及功能相关章节。
- 能按固定格式输出事件表：表名、功能描述、通用表头、字段类型、字段说明、负责策划。
- 能在文末给出为什么需要这些打点，以及后续用哪些数据判断优化方向。
- 原生页签不可用时，能明确说明并安全回退为 `## 数据打点` 章节。
