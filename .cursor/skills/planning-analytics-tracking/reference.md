# 策划案数据打点参考

## 参考表名格式

表名用于表达“哪个业务、哪个模块、在哪个生命周期动作上报”。

推荐规则：

- 字符风格：全小写英文或团队约定缩写，默认不使用下划线。
- 禁止：中文、空格、临时拼音、只有数字或含义不明的缩写。
- 推荐结构：`{业务域或玩法}{对象或模块}{生命周期动作}`。
- 示例动作：`enter`、`show`、`click`、`select`、`start`、`end`、`settle`、`result`、`fail`、`reward`。
- 如果团队已有前缀规则（如 `fact_`、`dwd_`、`lc_`），优先沿用。
- 新表名需要查重；不确定时输出候选表名并标记 `待数据/后台确认`。

示例：

| 场景 | 推荐表名 |
| --- | --- |
| 小队副本战斗结束 | `raidbattleend` |
| 小队副本入口曝光 | `raidentryshow` |
| 小队副本阵容确认 | `raidteamsubmit` |
| 奖励领取 | `raidrewardclaim` |

## 交付结构

采用“总览不变，先判断承接，再选择明细”的结构交付。不要再输出所有事件共用的一张横向大表，也不要把设置项、开关、滑条、多档选择机械拆成多张事件表；同样也不要把教程、loading、点击、结算等已有承接事件硬压成轻量映射表。

```markdown
# 数据打点需求

| 说明 | 本页仅保留功能行为字段；player_id/user_id、server_id/zone_id、event_time、platform、app_version、session_id 等基础公共维度由登录、用户维表或埋点 SDK 统一提供，不在每张功能表重复设计。 |
| 负责策划 | <姓名或待策划确认> |

行数不固定，按实际指标完整提炼；不要因为参考格式有 3 行就只写 3 行。相近问题可以合并，但不能为了压缩行数漏掉关键链路。通常覆盖入口参与、流程转化、核心玩法结果、奖励消耗、社交反馈、异常/失败等主要复盘方向。

| --- | --- |
| 上报端 | <客户端 / 服务器 / 待确认> |

## <轻量映射表名>

适用于设置项、配置项、功能列表、push 类型、教程步骤、UI 点击内容等枚举说明。只有运行时行为已由统一承接事件表记录，或本次只是补充枚举维表时，才单独输出。

| change_type | <content_key> | 中文分类 | 备注 | UI类型 |
| --- | --- | --- | --- | --- |
| <模块或事件归类> | <配置项/枚举项 key> | <中文大类> | <含义/默认值/限制说明> | <开关/多档选择/按钮/外链/拦截等> |

## <复杂事件 table_name>

| 表名 | <table_name> |
| 功能描述 | <功能描述> |
| 通用表头 | 字段类型 | 字段说明 |  | 必要性 | 字段备注 |
| <field_name> | string | <description> |  | <why_needed> | <enum_or_note> |
```

轻量映射表固定列说明：

| 列名 | 含义 |
| --- | --- |
| change_type | 事件或模块归类，例如 `setting_screen`、`setting_audio` |
| content_key | 具体配置项或枚举项 key，可按业务替换为 `setting_content`、`unlock_content` 等 |
| 中文分类 | 策划和分析阅读的大类 |
| 备注 | 具体含义、默认值、限制、待确认说明 |
| UI类型 | 控件或交互形态，例如开关、多档选择、音量滑动、按钮、外链、拦截 |

固定列说明：

| 列名 | 含义 |
| --- | --- |
| 通用表头 | 字段名，例如 `raidboss_id` |
| 字段类型 | 只填写 `string` |
| 字段说明 | 字段用途、枚举含义、单位、来源表、复合字段格式 |
| 必要性 | 为什么需要这个字段；若只是补充分析则写“可选” |
| 字段备注 | 枚举映射、版本限制、配置来源、暂不开放说明 |

负责人只在页签顶部写一次，不在每个字段行重复。

## 需求提炼边界

生成的是策划侧“打点需求稿”，不是数据组后续埋点方案。不要把需求稿写成最终数据库方案。

需求稿需要补充：

- `需求描述总览`：放在格式最前面，按 `功能名 / 需求描述 / 目的` 汇总本次打点要回答的指标问题；示例只代表格式，内容必须从当前文档提炼，行数不固定。
- `结构判断`：必须先判断上报端、承接方式、记录粒度和明细结构，再开始写表。优先匹配参考埋点设计表中的已有承接表。
- 明细按承接方式输出：已有承接事件表时输出事件字段块，枚举维表作为补充；只有纯配置/枚举补充时才只输出轻量映射表。
- 不新增 `记录时机`、`打点侧建议`、`关联说明` 等数据组方案字段，除非用户明确要求。

字段筛选：

- 明确服务核心指标的字段写为必需。
- 只是“可能有用/方便分析”的字段写为可选或待确认。
- 同一行为能通过 `action`、`status`、`type` 类字段扩展枚举表达时，优先建议扩展枚举，不新增多个含义相近字段。
- 待确认事项精简到 1 到 3 个关键问题。

## 用户修订后的问题总结

旧输出需要避免的问题：

- 把所有表放进同一张横向大表，阅读成本高，不符合策划审阅习惯。
- 每行重复 `负责策划`，造成冗余；负责人应放在页签顶部。
- 输出 `字段归属` 列过于工程化，对策划审阅价值有限；保留在内部判断，不作为交付列。
- 将 `player_id`、`server_id`、`event_time` 等公共维度混入功能表，导致每张表重复基础字段。
- 机械拆分展示、点击、结果表，导致表数量过多；同一复盘目标下可以合并。
- 对未开放/留坑功能设计完整链路字段，实际只需要点击结果和报错。
- 枚举说明散落在字段说明中，应该优先放入 `字段备注`，例如 `1-大地图、2-排行榜、3-组队副本、4-IM`。
- 设置类需求被拆成 `uiaction/change/service/push/configerror` 等多张表，字段过重；应先判断是否复用 `lgrolesettingchange`/`settingchange`，再输出事件字段或轻量映射。
- 教程、loading、UI 点击等需求被误压成自造轻量表，导致无法落到现有埋点设计；应先判断是否复用 `lgrolefte`、`lgroleloading`、`lgroleuiclick` 等承接表，再补充 `FTE步骤`、`loading步骤`、`uiclick` 这类枚举维表。

## 参考埋点设计表结构判断

参考设计表不是单一模板，而是“主承接表 + 枚举/步骤维表”的组合：

- `客户端打点设计` / `服务器打点设计`：主设计页。顶部是通用表头，中部用 `大类 / 模块 / 表名 / 功能描述` 做承接表目录，后续每张表用 `新增表 / 表名 / 功能描述 / 字段名 / 字段类型 / 字段说明` 展开字段。
- `FTE步骤`、`loading步骤`、`settingchange`、`uiclick`：枚举、步骤或内容维表，用于解释事件表中的 key、type、content、step，不替代事件表。
- `优先级 / 程序对接人 / 程序进度 / 预计完成时间 / 数据组 / 日志平台建表 / QA / 验证结果 / 造数据需求 / QA备注 / 上线状态`：落地跟踪列，不作为策划侧打点需求字段。

常见承接判断：

| 需求类型 | 优先承接 | 明细建议 |
| --- | --- | --- |
| FTE/教程步骤 | `lgrolefte` + `FTE步骤` | 事件字段块记录步骤发生/完成/失败等运行时字段；维表记录 action/step 枚举 |
| loading/登录流程流失 | `lgroleloading` + `loading步骤` | 事件字段块记录 loading key、阶段、是否通过、耗时等；维表解释步骤 |
| UI 点击 | `lgroleuiclick` + `uiclick` | 若只补点击入口，输出 uiclick 枚举；若需点击结果/失败原因，补事件字段 |
| 设置变更 | `lgrolesettingchange` + `settingchange`，必要时 `lgrolesettingsnap` | 设置项枚举放维表；每次变更或登录快照由承接表记录 |
| 战斗/玩法结算 | 服务器玩法表，如 `lgroleclimbtower`、`lgroledojo`、`lgroleraidbossbattles` | 输出事件字段块，保留关卡、结果、耗时、阵容、伤害、奖励等上下文 |
| 道具/资源/奖励 | 服务器流动表，如 `lgroleitemflow`、`lgroleresflow`、活动奖励表 | 输出事件字段块，记录 action、channel、item_id、数量、余额、来源 |

## 二次询问基准

以下不确定事项必须先问用户，不要直接生成最终版：

- 文档包含多个功能页签，不确定本次是否全部纳入。
- 某字段可能由公共埋点头、登录表、用户维表或其他专表提供，不确定是否放进功能表。
- 关键口径未定且会影响表结构，例如队伍人数、战斗明细是否由战斗组专表承接、奖励系统是否单独出表。
- 已有 `数据打点` 页签，不确定是追加、覆盖还是另建。
- 文档多人协作，正式交付需要指定负责策划。
- 字段或逻辑只来自黑名单页签/章节，无法确认是否为最终实现。
- 详细方案中的截图、长图、流程图或画板承载核心交互/状态/流程，但当前无法稳定读取图片内容。

不阻塞但要写入备注的事项：

- 配置值待定，例如奖励数量 x、每日上限 X、默认 150s。
- 复合字段也填写 `string`，具体结构写在字段说明或字段备注中。
- 七日不做/留坑功能，只记录必要点击与结果。

## 图片与截图阅读基准

详细方案中的图片、长图、界面示意和流程图可能是功能理解的主要来源。生成打点需求前必须先判断是否需要读图。

必须读图的情况：

- 文字只写了标题或摘要，但图片里包含页面入口、按钮、弹窗、状态、流程节点。
- 表格读取结果大量为空，但用户实际文档中存在截图、长图、附件或画板。
- UI 截图中展示了成功、失败、未开放、冷却、奖励、消耗、数量限制等可影响字段设计的信息。
- 流程图中展示了步骤分支、异常分支或结果反馈。

处理规则：

- 先用可用的飞书下载、导出或用户附件获取图片，再进行视觉理解。
- 如果图片无法稳定识别，先二次询问，不要只根据文字补全。
- 图片中属于 `美术需求`、视觉包装、动效、素材占位的内容仍不进入功能打点。
- 输出时只保留服务核心复盘问题的字段，不因图片信息变多而扩表堆字段。

## 读取黑名单

以下页签或章节默认不参与功能打点需求提取：

- `美术需求`
- `美术`
- `视觉需求`
- `表现需求`
- `资源需求`
- `动效需求`
- `音频需求`
- `素材需求`

原因：

- 这些内容通常是中间交付，不代表最终功能实现。
- 其中的资源、表现、包装、临时占位容易被误读为功能字段。
- 如果主功能页签和美术需求冲突，以主功能/详细方案/规则/结算/奖励等实现口径为准。

例外：

- 用户明确要求分析美术、资源、动效或音频相关打点时可以读取。
- 读取时必须在输出中注明这是“表现/资源口径”，不是功能实现口径。
