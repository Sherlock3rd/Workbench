# 数据打点输出格式冗余

## 基本信息
- 日期：2026-05-11
- 分类：rules
- 任务：根据飞书策划案生成数据打点需求

## 问题现象
- 初版数据打点页签使用单张横向大表，字段过多，阅读成本高。
- 每行重复 `负责策划`，造成无效冗余。
- 输出了 `字段归属` 列，偏工程分析过程，不适合作为策划交付列。
- 将 `player_id`、`server_id`、`event_time` 等公共维度混入功能表，导致功能打点与基础行为/基础维度边界不清。
- 对部分留坑或未开放功能设计了过完整的字段链路。

## 原因分析
- 生成时过度追求“字段完整”，没有先判断字段是否服务当前功能复盘问题。
- 没有区分公共埋点维度、用户基础维表字段和功能行为字段。
- 没有遵循用户更适合审阅的“表名块 + 字段小表”格式。

## 修复动作
- 将 Skill 输出基准改为：页签顶部说明与负责策划；公共维度单独成块；每个事件表独立块呈现。
- 字段明细列改为：通用表头、字段类型、字段说明、必要性、字段备注。
- 删除默认输出中的 `字段归属` 和逐字段 `负责策划`。
- 在 Skill 中增加合并规则：同一复盘目标下，展示、点击、未开放结果可合并；留坑功能只记录点击和结果。

## 预防办法
- 先判断“这张表后续是否会被单独看指标”，再决定是否拆表。
- 先判断“这个字段是否由公共维度/用户维表/SDK 已提供”，再决定是否写入功能表。
- 枚举、版本限制、配置来源优先写入 `字段备注`。

## 执行前检查项
- [ ] 是否已把公共维度与功能字段分离
- [ ] 是否避免了横向大表
- [ ] 是否移除了逐行重复的负责策划
- [ ] 是否为每个字段写明必要性
- [ ] 是否只为留坑/未开放功能保留必要点击与结果字段
