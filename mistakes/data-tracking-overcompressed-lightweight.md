# 数据打点过度轻量化

## 基本信息
- 日期：2026-05-15
- 分类：rules
- 任务：根据飞书策划案生成数据打点需求

## 问题现象
- 根据设置类反馈优化后，后续需求被机械套用轻量映射表。
- 教程、loading、UI 点击等本应先匹配现有承接事件表的需求，被压缩成自造的 `change_type / content_key / 中文分类 / 备注 / UI类型` 表。
- 结果缺少上报端、承接表、记录粒度、运行时字段和与现有埋点设计表的对应关系，导致方案不可落地。

## 原因分析
- 把“设置类可以轻量化”误扩展成“枚举类都只输出轻量表”。
- 没有先参考现有 `客户端打点设计`、`服务器打点设计`、`FTE步骤`、`settingchange`、`uiclick` 等表结构。
- 没有区分“主事件承接表”和“枚举/步骤维表”的职责。

## 修复动作
- 在 `planning-analytics-tracking` Skill 中新增“埋点设计参考表判断流程”。
- 输出中增加 `## 结构判断`，先说明上报端、承接方式、记录粒度、明细结构和判断理由。
- 明确轻量映射表只能作为枚举/配置说明；已有承接事件表时，应输出事件字段块并按需补充枚举维表。

## 预防办法
- 生成打点前先判断是否复用已有承接表，例如 `lgrolefte`、`lgroleloading`、`lgroleuiclick`、`lgrolesettingchange`。
- 只有纯配置/枚举补充且运行时行为已由统一事件承接时，才只输出轻量映射表。
- 教程、loading、玩法结算、资源流动、奖励、任务进度等必须先判断记录粒度和运行时字段。

## 执行前检查项
- [ ] 是否先匹配现有客户端/服务器承接表
- [ ] 是否说明上报端、承接方式、记录粒度和明细结构
- [ ] 是否区分主事件表与枚举/步骤维表
- [ ] 是否避免自造无法落地的轻量表
